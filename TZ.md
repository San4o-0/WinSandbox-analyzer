# Технічне завдання: FileChecker — перевірка файлів у віртуальному середовищі Windows

## 1. Мета

Десктопний застосунок для Windows, який дозволяє користувачу перетягнути (drag & drop) або обрати файл, після чого файл автоматично аналізується в ізольованому віртуальному середовищі (Windows Sandbox), а користувач отримує звіт з вердиктом про потенційну небезпечність файлу. Файл ніколи не запускається на хостовій системі.

## 2. Цільова платформа та вимоги до системи

- ОС: Windows 10/11 Pro, Enterprise або Education (редакції з підтримкою Windows Sandbox).
- Увімкнений компонент "Windows Sandbox" (`Enable-WindowsOptionalFeature -FeatureName "Containers-DisposableClientVM"`).
- Апаратна віртуалізація (VT-x/AMD-V) увімкнена в BIOS/UEFI.
- Python 3.10+ (для запуску застосунку).
- Мінімум 4 ГБ вільної RAM (пісочниця споживає ~2 ГБ).

## 3. Загальна архітектура

```
+---------------------------+
|  GUI (Tkinter)            |
|  drag&drop / вибір файлу  |
+------------+--------------+
             |
+------------v--------------+
|  Статичний аналіз (хост)  |
|  хеші, ентропія, PE,      |
|  підозрілі рядки          |
+------------+--------------+
             |
+------------v--------------+
|  SandboxManager           |
|  - сесійна тека           |
|  - генерація .wsb         |
|  - запуск WindowsSandbox  |
|  - очікування звіту       |
+------------+--------------+
             |
+------------v--------------+
|  Windows Sandbox (VM)     |
|  analyzer.ps1:            |
|  - знімок системи "до"    |
|  - запуск файлу           |
|  - знімок системи "після" |
|  - diff -> report.json    |
+------------+--------------+
             |
+------------v--------------+
|  Звіт + вердикт (GUI)     |
+---------------------------+
```

## 4. Функціональні вимоги

### 4.1. Приймання файлу
- Drag & drop файлу у вікно застосунку.
- Альтернатива: кнопка "Обрати файл" зі стандартним діалогом.
- Одночасно обробляється один файл; черга наступних відображається у списку.
- Максимальний розмір файлу: 500 МБ (конфігурується).

### 4.2. Статичний аналіз (на хості, без запуску файлу)
- Обчислення SHA-256 та MD5.
- Обчислення ентропії Шеннона (висока ентропія → ознака пакування/шифрування).
- Визначення типу файлу за сигнатурою (magic bytes): PE (MZ), скрипти, архіви, документи.
- Виявлення розбіжності розширення і реального типу (наприклад, `.pdf`, що є PE).
- Пошук підозрілих рядків: імена WinAPI (VirtualAlloc, CreateRemoteThread, WriteProcessMemory тощо), URL, PowerShell-патерни, base64-блоки.
- Виявлення подвійних розширень (`invoice.pdf.exe`).

### 4.3. Динамічний аналіз (у Windows Sandbox)
- Створення одноразової сесійної теки `%USERPROFILE%\FileChecker\sessions\<id>\shared`, куди копіюються файл та `analyzer.ps1`.
- Генерація конфігурації `.wsb`: мапінг сесійної теки, вимкнений доступ до мережі (конфігурується), обмеження RAM, автозапуск аналізатора через LogonCommand.
- Дії `analyzer.ps1` всередині пісочниці:
  1. Знімок стану "до": процеси, служби, заплановані завдання, ключі автозапуску реєстру, файли у критичних теках, мережеві з'єднання.
  2. Запуск досліджуваного файлу відповідно до типу (exe — напряму; скрипти — інтерпретатором; документи — асоційованим застосунком).
  3. Спостереження протягом заданого часу (за замовчуванням 60 с).
  4. Знімок стану "після" та обчислення різниці.
  5. Запис `report.json` у змапену теку.
- Хост очікує `report.json` з таймаутом (за замовчуванням 5 хв), після чого пісочниця закривається, середовище знищується.

### 4.4. Вердикт
Бальна система (0–100):
- нові процеси, породжені файлом — до 15 балів;
- зміни автозапуску/реєстру — до 20 балів;
- нові заплановані завдання/служби — до 20 балів;
- нові виконувані файли у системних теках — до 15 балів;
- нові мережеві з'єднання — до 10 балів;
- статичні індикатори (ентропія, розбіжність типу, підозрілі рядки) — до 20 балів.

Рівні: 0–19 — «Чистий», 20–49 — «Підозрілий», 50–100 — «Небезпечний».

### 4.5. Звіт
- Відображення у GUI: вердикт, бал, хеші, тип файлу, список індикаторів, деталі динамічного аналізу.
- Збереження звіту як JSON у сесійній теці; можливість експорту.
- Історія перевірок у межах сеансу застосунку.

## 5. Нефункціональні вимоги

- Мова інтерфейсу: українська.
- Файл ніколи не виконується на хості; будь-який запуск — лише всередині пісочниці.
- Після завершення аналізу пісочниця знищується разом з усіма змінами (властивість Windows Sandbox).
- Мережа в пісочниці вимкнена за замовчуванням.
- Логування дій застосунку в `%USERPROFILE%\FileChecker\filechecker.log`.
- Аналіз не блокує інтерфейс (виконання у фоновому потоці).

## 6. Обмеження

- Одночасно може працювати лише один екземпляр Windows Sandbox — черга обробляється послідовно.
- Динамічний аналіз обмежений можливостями PowerShell-моніторингу (без kernel-драйвера); шкідливе ПЗ з анти-VM-технікою може не проявити активність.
- Застосунок є інструментом первинної оцінки і не замінює повноцінний антивірус.

## 7. Структура проєкту

```
FileChecker/
├── TZ.md
├── README.md
├── requirements.txt
├── main.py              GUI, черга, історія
├── static_analysis.py   статичний аналіз файлу
├── sandbox_manager.py   сесії, .wsb, запуск пісочниці, збір звіту
├── verdict.py           бальна система і вердикт
├── config.py            налаштування
└── sandbox/
    └── analyzer.ps1     скрипт динамічного аналізу в пісочниці
```

## 8. Сценарій використання

1. Користувач запускає `python main.py`.
2. Перетягує файл у вікно.
3. Застосунок виконує статичний аналіз і одразу показує попередні індикатори.
4. Автоматично формується сесія, запускається Windows Sandbox, файл виконується всередині.
5. Через 1–2 хвилини у вікні з'являється повний звіт з вердиктом.
6. Пісочниця закривається, всі зміни в ній знищуються.
